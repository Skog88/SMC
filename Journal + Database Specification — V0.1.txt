Journal + Database Specification — V0.1

The Journal is the most important long-term component of your system.

The Rule Engine finds the setup.
The AI Brain scores the setup.
The Execution Engine places the trade.
But the Journal tells you whether the system is actually improving.

The PDF strategy depends on a strict sequence: M15 direction and structure, weakness sweep, rectangle creation, M1 flip, then entry with SL/TP. The journal must log every stage of that sequence, because later we need to know exactly where the edge exists and where it fails. The PDF also emphasizes that M15 identifies the setup and M1 decides the entry, so both timeframes must be logged separately.

1. Core Principle

The Journal must log setups, not only trades.

A normal trading journal logs this:

Entry
SL
TP
Exit
Profit/loss

Your system needs more:

Why was the setup detected?
Why was it skipped?
What did Claude score it?
Did M1 confirm?
Did spread block it?
Did it reach TP/SL?
Was the AI score actually predictive?

That means the main database object is not the trade.

The main object is:

setup_id

A trade may or may not happen from that setup.

4. Database Design

Use SQLite first.

Reason:

Local
Simple
Reliable
Easy backup
Easy CSV export
No server required
Perfect for one local MT5 system

Later, you can migrate to PostgreSQL if you run multiple terminals, multiple PCs, or cloud dashboards.

5. Table: setups

This is the core table.

It logs every mechanically detected setup, whether traded or not.

CREATE TABLE IF NOT EXISTS setups (
    setup_id TEXT PRIMARY KEY,

    symbol TEXT NOT NULL,
    direction TEXT NOT NULL,
    setup_time TEXT NOT NULL,
    setup_date TEXT NOT NULL,
    session TEXT,

    timeframe_setup TEXT DEFAULT 'M15',
    timeframe_entry TEXT DEFAULT 'M1',

    state TEXT NOT NULL,
    final_status TEXT,

    -- EMA / trend context
    ema_primary_period INTEGER,
    ema_secondary_period INTEGER,
    ema_primary_value REAL,
    ema_secondary_value REAL,
    price_vs_ema TEXT,
    ema_slope_points REAL,
    trend_direction TEXT,
    trend_valid INTEGER,

    -- Structure context
    marked_level_type TEXT,
    marked_level_price REAL,
    marked_level_time TEXT,
    clean_structure INTEGER,
    structure_score INTEGER,
    level_age_candles INTEGER,
    level_already_swept INTEGER,

    -- Imbalance / FVG context
    inside_imbalance INTEGER,
    imbalance_type TEXT,
    imbalance_low REAL,
    imbalance_high REAL,
    imbalance_distance_points REAL,

    -- Session level context
    level_is_session_high_low INTEGER,
    related_session TEXT,
    related_session_level TEXT,
    session_level_distance_points REAL,

    -- M15 trigger candle
    m15_trigger_time TEXT,
    m15_open REAL,
    m15_high REAL,
    m15_low REAL,
    m15_close REAL,
    m15_tick_volume INTEGER,

    -- Sweep details
    sweep_valid INTEGER,
    sweep_depth_points REAL,
    wick_size_points REAL,
    wick_to_range_ratio REAL,
    close_position REAL,

    -- Rectangle
    rectangle_low REAL,
    rectangle_high REAL,
    rectangle_size_points REAL,

    -- AI result summary
    ai_enabled INTEGER,
    ai_score INTEGER,
    ai_decision TEXT,
    ai_model TEXT,
    ai_prompt_version TEXT,

    -- M1 entry status
    m1_flip_confirmed INTEGER DEFAULT 0,
    m1_flip_time TEXT,
    m1_flip_close REAL,
    m1_wait_candles INTEGER,

    -- Risk/execution status
    risk_approved INTEGER,
    risk_block_reason TEXT,
    execution_attempted INTEGER DEFAULT 0,
    order_sent INTEGER DEFAULT 0,

    -- Skip/reject reason
    skip_reason TEXT,
    skip_stage TEXT,

    -- Timestamps
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

7. Table: ai_reviews

This table stores Claude’s full response.

Do not store only the score in setups. Store the full reasoning separately.

CREATE TABLE IF NOT EXISTS ai_reviews (
    review_id TEXT PRIMARY KEY,
    setup_id TEXT NOT NULL,

    provider TEXT,
    model TEXT,
    prompt_version TEXT,

    input_json TEXT NOT NULL,
    raw_response TEXT,
    parsed_json TEXT,

    confidence_score INTEGER,
    decision TEXT,
    reasoning TEXT,

    response_valid INTEGER,
    error_message TEXT,
    latency_ms INTEGER,

    created_at TEXT NOT NULL,

    FOREIGN KEY (setup_id) REFERENCES setups(setup_id)
);

Why this matters:

Later you can compare Claude prompt versions.
You can test if one scoring model was better than another.
You can fine-tune or train using the original setup JSON + actual result.

Useful event_type values:

M1_WAITING
M1_TOUCHED_RECTANGLE

14. Setup Lifecycle Logging

Every setup should move through this lifecycle:

1. SCANNING
2. M15_LEVEL_FOUND
3. M15_WEAKNESS_CONFIRMED
4. RECTANGLE_CREATED
5. AI_REVIEW_REQUESTED
6. AI_APPROVED or AI_REJECTED
7. M1_MONITORING_STARTED
8. M1_FLIP_CONFIRMED or M1_FLIP_TIMEOUT
9. RISK_APPROVED or RISK_BLOCKED
10. ORDER_SENT or ORDER_BLOCKED
11. TRADE_OPENED
12. TRADE_CLOSED
13. JOURNAL_COMPLETE

Each lifecycle update should update:

setups.state
setups.final_status
setups.updated_at
15. Setup ID Format

Use deterministic setup IDs.

{symbol}_{date}_{m15_time}_{direction}_{marked_level_price}

Example:

XAUUSD_2026-05-02_0945_LONG_2321.30

Safer version:

XAUUSD_20260502T094500_LONG_232130

This prevents duplicate setup logging if the system restarts.

16. Trade ID Format

Use:

{setup_id}_TRADE_{mt5_position_id}

CREATE VIEW IF NOT EXISTS ai_training_view AS
SELECT
    s.setup_id,
    s.symbol,
    s.session,
    s.direction,
    s.trend_valid,
    s.ema_slope_points,
    s.structure_score,
    s.inside_imbalance,

    s.wick_to_range_ratio,
    s.close_position,
    s.rectangle_size_points,

    s.m1_flip_confirmed,
    CASE WHEN t.trade_id IS NOT NULL THEN 1 ELSE 0 END AS trade_taken,
    t.pnl_r,
    s.final_status
FROM setups s
LEFT JOIN trades t ON s.setup_id = t.setup_id;

Keep other modules from writing SQL directly.

24. Journal Safety Rules

The journal must be resilient.

Rules:

Never lose a setup because one field is missing.
Never crash trading because screenshot failed.
Never block emergency trade close because database write failed.
Always log raw JSON if structured parsing fails.
Use transactions for trade creation/update.
Back up the SQLite file daily.

For write failures:

Write emergency backup JSONL file in /data/emergency_logs/
Continue system if safe.
Stop system only if trade state cannot be trusted.
25. Emergency JSONL Backup

Create:

data/emergency_logs/journal_fallback_2026-05-02.jsonl

Each line:

{"time":"2026-05-02T09:45:01+02:00","event":"SETUP_CREATED","payload":{...}}

This protects you if SQLite locks or corrupts.

27. What the Feedback Loop Actually Means

Important: the AI does not “learn” automatically just because the journal exists.

The real feedback loop is:

1. Journal collects setup + AI + outcome data
2. Reports identify patterns
3. You adjust:
   - AI prompt
   - AI score threshold
   - rule filters
   - session filters
   - symbol filters
4. Later, after enough data, you can train/fine-tune a model

For the first version, the feedback loop is analytics-driven, not self-training.

29. Final Journal Decision Flow
Rule Engine sees possible setup
        ↓
Journal logs setup attempt
        ↓
Mechanical rules confirm or reject
        ↓
If rejected → log skip reason
        ↓
If valid → log rectangle
        ↓
AI reviews setup
        ↓
Journal logs score + reasoning
        ↓
If AI rejects → final_status = AI_REJECTED
        ↓
If AI approves → monitor M1
        ↓
Journal logs M1 behavior
        ↓
If M1 flips → risk check
        ↓
If risk passes → trade created
        ↓
Execution events logged
        ↓
Trade closes
        ↓
Journal logs final result
        ↓
Reports update feedback loop
